# PRD: Slate OS v0.0.7 — Core Applications

**Version:** v0.0.7
**Status:** Draft
**Author:** Adam
**Last Updated:** 2026-07-03

---

# Overview

Slate OS v0.0.7 introduces the first generation of native Slate applications.

Previous releases established the desktop environment and system configuration.

This milestone begins replacing generic Linux applications with applications designed specifically for Slate OS.

The objective is consistency rather than feature completeness.

---

# Objective

Provide a minimal set of native applications required for daily desktop usage.

Applications should share:

- visual language
- navigation patterns
- typography
- spacing
- keyboard shortcuts
- window behavior

---

# Problem Statement

Although the desktop environment now feels like Slate, users still interact with third-party applications that differ significantly in appearance and behavior.

This creates an inconsistent user experience.

Native applications establish Slate as a cohesive platform rather than a customized Linux distribution.

# Functional Requirements

## FR-001 — Slate Terminal

Provide a native terminal application.

Minimum functionality:

- execute shell commands
- multiple tabs
- copy & paste
- configurable font size

---

## FR-002 — Slate Files

Provide a native file manager.

Minimum functionality:

- browse folders
- create folders
- rename files
- delete files
- copy & paste
- drag & drop

Network locations are out of scope.

---

## FR-003 — Slate Calculator

Provide a basic calculator.

Supported operations:

- addition
- subtraction
- multiplication
- division
- percentage

Scientific mode is out of scope.

---

## FR-004 — Slate Image Viewer

Provide a lightweight image viewer.

Supported formats:

- PNG
- JPG
- WebP

Editing is out of scope.

---

## FR-005 — Shared Design System

All applications shall use the same:

- colors
- typography
- spacing
- icons
- buttons
- dialogs

---

## FR-006 — Launcher Integration

All native applications shall appear in the Slate Launcher.

---

## FR-007 — Window Consistency

All applications shall support:

- minimize
- maximize
- close
- resizing

---

# Non-Functional Requirements

## Startup Performance

Applications should launch within approximately one second on supported hardware.

---

## Reliability

Applications must not crash during normal usage.

---

## Maintainability

Applications should reuse shared Slate UI components whenever possible.

---

# Out of Scope

- Office suite
- Browser
- Mail client
- Music player
- Video player
- PDF editor
- Text editor
- Package manager
- Store
- AI assistant
- Cloud synchronization
- Notes
- Calendar
- Contacts

---

# Design Principles

Applications should feel like members of the same family.

Consistency is more important than feature count.

Avoid implementing advanced functionality before core workflows are complete.

---

# Success Criteria

The milestone is complete when:

- Slate Terminal launches and functions correctly.
- Slate Files performs basic file operations.
- Slate Calculator performs basic arithmetic.
- Slate Image Viewer opens supported image formats.
- All applications follow the Slate Design System.
- All applications appear in the Launcher.

---

# Failure Criteria

The milestone fails if:

- Applications crash.
- UI is inconsistent.
- Shared components diverge.
- Applications fail to launch.

---

# Validation Procedure

1. Launch every Slate application.
2. Verify visual consistency.
3. Perform basic workflows.
4. Close applications.
5. Reopen applications.
6. Verify Launcher integration.
7. Repeat testing across multiple sessions.

---

# Risks

| Risk | Mitigation |
|------|------------|
| Duplicating existing Linux software | Focus on user experience rather than feature parity |
| Inconsistent UI | Build reusable component library |
| Scope creep | Implement only essential functionality |
| Maintenance burden | Share code across applications |
