# PRD: Slate OS v0.0.5 — Authentication & Session Management

**Version:** v0.0.5
**Status:** Draft
**Author:** Adam
**Last Updated:** 2026-07-03

---

# Overview

Slate OS v0.0.5 introduces the first complete user session workflow.

Previous milestones focused on creating a working desktop environment.

This release defines how users enter, leave and manage desktop sessions.

Authentication becomes a first-class component of Slate OS.

---

# Objective

Implement a secure and reliable session lifecycle.

Users should be able to:

- log in
- lock the session
- unlock the session
- log out
- restart
- power off

---

# Problem Statement

The current system starts directly into the desktop.

While sufficient for development, this is unsuitable for public use.

Authentication and session management are required before Slate OS can move toward external testing.

# Goals

- Introduce authentication.
- Manage user sessions.
- Support secure desktop locking.
- Support graceful logout.
- Prepare for future multi-user support.

---

# Functional Requirements

## FR-001 Login Screen

Display the Slate Login screen before starting the desktop session.

---

## FR-002 Password Authentication

Users authenticate using a password.

No biometric authentication is required.

---

## FR-003 Automatic Session Start

After successful authentication:

- desktop launches
- user session starts
- desktop becomes interactive

---

## FR-004 Invalid Password Handling

Incorrect passwords display an error.

No desktop session starts.

---

## FR-005 Session Lock

Users can lock the current session.

Unlocking requires authentication.

---

## FR-006 Logout

Users can end the current session.

Desktop returns to Login.

---

## FR-007 Restart

Restart from Login Screen.

Restart from Desktop.

---

## FR-008 Shutdown

Power off from Login Screen.

Power off from Desktop.

---

## FR-009 Session Recovery

Unexpected desktop crashes return to Login instead of a black screen.

---

# Non Functional Requirements

## Security

Passwords must never be stored in plaintext.

---

## Stability

Authentication failures must never crash the desktop.

---

## Reliability

Session transitions should complete without graphical glitches.

---

# Out of Scope

- Multiple users
- Fingerprint login
- Face ID
- PIN
- Password recovery
- Online accounts
- Child accounts
- Permissions
- User avatars
- Cloud sync

---

# Success Criteria

✓ Login works

✓ Wrong password handled correctly

✓ Lock screen works

✓ Logout works

✓ Restart works

✓ Shutdown works

✓ Session survives repeated testing

---

# Failure Criteria

- Authentication bypass
- Black screen
- Frozen login
- Desktop crash after login
- Logout failure
- Session corruption

---

# Validation Procedure

1 Boot system

2 Login

3 Lock desktop

4 Unlock

5 Logout

6 Login again

7 Restart

8 Login

9 Shutdown

Repeat multiple times.

---

# Risks

| Risk | Mitigation |
|------|------------|
| PAM configuration | Use standard Linux authentication |
| Session crashes | Return safely to Login |
| Authentication bugs | Extensive repeated testing |
| Scope creep | No multi-user support |
