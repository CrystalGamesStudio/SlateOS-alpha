# Overview

Slate OS v0.0.1 is the first functional milestone of the project.

The objective of this release is **not** to build a usable operating system. The objective is to validate the most critical technical assumption of the entire project:

> Slate OS can boot into a stable graphical desktop environment.

The previous attempt at building Slate failed because the graphical environment never reached a reliable, usable state. Before investing time into branding, applications, login systems or compatibility layers, this milestone must prove that the graphical foundation is stable.

Everything not directly contributing to this validation is intentionally excluded.

---

# Problem Statement

The largest technical risk of Slate OS is the graphical environment.

A working terminal is not sufficient.

A working kernel is not sufficient.

A booting Linux installation is not sufficient.

The project can only continue if Slate reliably launches a graphical desktop environment without crashes or rendering failures.

This release exists solely to answer that question.

---

# Goals

The release should achieve the following goals:

- Boot successfully inside QEMU.
- Automatically start the graphical session.
- Render the desktop correctly.
- Allow opening and closing application windows.
- Run without crashes during normal operation.

# Functional Requirements

## FR-001

The system shall boot successfully inside QEMU.

---

## FR-002

The graphical session shall start automatically after boot.

No login screen is required.

---

## FR-003

The desktop background shall render correctly.

A temporary static Slate wallpaper is sufficient.

---

## FR-004

The system shall include a working terminal emulator.

The terminal must:

- launch successfully
- receive keyboard input
- execute commands
- close normally

---

## FR-005

Windows shall be movable.

---

## FR-006

Windows shall be resizable.

---

## FR-007

Windows shall be closable.

---

# Non-Functional Requirements

## Stability

The system must complete ten consecutive boots without graphical failures.

---

## Reliability

No compositor crash is acceptable during basic testing.

---

## Performance

Desktop should become usable within approximately 15 seconds inside QEMU.

Exact optimization is out of scope.

---

## Maintainability

Configuration should remain minimal.

Avoid unnecessary custom code.

Prefer upstream defaults whenever possible.

# Success Criteria

The milestone is considered complete only if all conditions are met.

- System boots successfully.
- GUI launches automatically.
- Desktop renders correctly.
- Terminal works.
- Windows can be opened.
- Windows can be moved.
- Windows can be resized.
- Windows can be closed.
- No compositor crashes occur.
- Ten consecutive successful boots are completed.

---

# Failure Criteria

The milestone is considered failed if any of the following occur:

- Black screen
- Frozen desktop
- Hyprland crash
- Terminal cannot launch
- Window management fails
- Desktop fails to render
- Manual intervention required during boot

---

# Validation Procedure

For every build:

1. Boot inside QEMU.
2. Wait until desktop appears.
3. Launch terminal.
4. Execute basic shell commands.
5. Resize window.
6. Move window.
7. Close window.
8. Repeat ten times.

Every run must complete successfully.

---

# Risks

| Risk | Mitigation |
|------|------------|
| Hyprland instability | Keep configuration minimal |
| QEMU graphics issues | Test using virtio-gpu |
| Wayland configuration errors | Use upstream defaults |
| Excessive customization | Postpone all visual polish |
